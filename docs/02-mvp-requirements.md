# FlowPilot MVP Requirements

## 1. 문서 목적

이 문서는 MVP 범위의 기능 요구사항을 구현 가능한 수준으로 구체화한다.
각 기능은 사용자 가치, 세부 요구사항, 수용 기준, 제외 범위를 포함한다.

---

## 2. 제품 목표

MVP는 사용자가 GitHub Actions 워크플로우를 연결하고, 구조를 시각적으로 이해하고, 최근 실행 결과를 확인하고, AI 기반 개선 포인트를 받는 경험을 제공해야 한다.

MVP 완료 기준:

1. 사용자가 Public 또는 Private GitHub 레포를 연결할 수 있다.
2. `.github/workflows` 아래 워크플로우를 목록으로 볼 수 있다.
3. 선택한 워크플로우를 Job DAG로 볼 수 있다.
4. 최근 실행 이력과 실패 Job을 확인할 수 있다.
5. AI 분석 결과를 심각도별로 볼 수 있다.

---

## 3. 사용자 스토리

### 3.1 US-01. 레포 연결

개발자로서, 내가 관리하는 GitHub 레포를 입력해서 워크플로우를 불러오고 싶다.

가치:

- 대상 레포를 바로 분석할 수 있다.

세부 요구사항:

- 사용자는 GitHub Username, PAT, 레포 URL을 입력할 수 있다.
- 앱은 레포 URL에서 owner/repo를 파싱해야 한다.
- 앱은 PAT 없이 public repo 조회를 허용할 수 있어야 한다.
- Private repo는 PAT가 없거나 권한이 부족하면 접근 실패를 명확히 표시해야 한다.
- 연결 성공 시 레포 메타정보를 메모리와 세션 스토리지에 저장할 수 있다.

수용 기준:

- 올바른 public repo URL 입력 시 워크플로우 목록이 로드된다.
- 올바른 private repo URL과 PAT 입력 시 워크플로우 목록이 로드된다.
- 잘못된 URL이면 입력 필드 아래에 검증 에러가 나온다.
- 401/403/404 오류는 서로 다른 메시지로 표시된다.

제외:

- GitHub OAuth
- 사용자 계정 시스템

---

### 3.2 US-02. 워크플로우 목록 조회

개발자로서, 레포 안의 워크플로우 파일 목록을 보고 분석할 파일을 선택하고 싶다.

가치:

- 여러 워크플로우 중 원하는 대상을 빠르게 선택할 수 있다.

세부 요구사항:

- 앱은 `.github/workflows` 경로를 조회한다.
- `.yml`, `.yaml` 파일만 목록에 포함한다.
- 목록은 파일명 기준 오름차순 정렬한다.
- 첫 로드 시 첫 번째 파일을 자동 선택한다.
- 선택 상태가 화면에 명확히 표시되어야 한다.

수용 기준:

- 워크플로우 파일이 1개 이상이면 목록이 표시된다.
- 워크플로우 파일이 없으면 empty state를 표시한다.
- 다른 파일 선택 시 DAG와 실행 이력 컨텍스트가 갱신된다.

제외:

- 폴더 트리 탐색
- 워크플로우 검색

---

### 3.3 US-03. Job DAG 시각화

개발자로서, 워크플로우의 Job 구조와 의존 관계를 그래프로 이해하고 싶다.

가치:

- 전체 실행 흐름을 YAML보다 빠르게 파악할 수 있다.

세부 요구사항:

- 앱은 YAML의 `jobs`를 파싱한다.
- 각 Job은 노드로 표현한다.
- `needs`는 방향 엣지로 표현한다.
- `needs`가 없는 Job은 시작 레벨에 배치한다.
- 병렬 가능 Job은 같은 레벨에 가깝게 배치한다.
- 노드에는 최소한 Job 이름, 상태, Step 수를 표시한다.
- 노드 클릭 시 JobDetail에 선택 정보가 반영된다.
- 그래프는 확대/축소/드래그가 가능해야 한다.

수용 기준:

- 단일 Job 워크플로우도 정상 렌더링된다.
- `needs` 배열과 문자열 형식을 모두 지원한다.
- 파싱 불가 YAML은 그래프 대신 오류 상태를 표시한다.
- 선택된 노드는 시각적으로 강조된다.

제외:

- Matrix strategy 시각화
- 조건 분기 if 시각화
- Step 단위 DAG

---

### 3.4 US-04. 최근 실행 이력 조회

개발자로서, 워크플로우의 최근 실행 결과를 함께 보고 싶다.

가치:

- 구조뿐 아니라 실제로 어디가 실패했는지 확인할 수 있다.

세부 요구사항:

- 앱은 최근 10건의 workflow runs를 조회한다.
- 실행 목록에는 상태, conclusion, 브랜치, 이벤트, 생성 시각을 표시한다.
- 사용자가 특정 run을 선택하면 해당 run의 jobs를 조회한다.
- run job 결과를 DAG 노드 상태에 매핑한다.
- 실패 Job이 있으면 그래프에서 눈에 띄게 표시한다.

수용 기준:

- 실행 이력이 없으면 empty state가 표시된다.
- run 선택 시 그래프 상태가 1초 이내에 갱신된다.
- 실패/성공/진행중 상태가 서로 다른 시각 요소로 구분된다.

제외:

- 실시간 폴링
- 로그 내용 직접 표시

---

### 3.5 US-05. Job 상세 보기

개발자로서, 특정 Job의 구성과 실행 결과를 자세히 보고 싶다.

가치:

- 실패 원인과 병목 Step을 더 빠르게 찾을 수 있다.

세부 요구사항:

- 선택된 Job의 이름, needs, runs-on, if, steps를 표시한다.
- run 선택 상태일 경우 각 Step의 실행 시간과 conclusion을 표시한다.
- 가장 오래 걸린 Step을 강조한다.
- GitHub의 job 또는 run 상세 링크를 제공한다.

수용 기준:

- 노드 선택 시 상세 패널이 즉시 갱신된다.
- 실행 정보가 없더라도 정적 YAML 정보는 표시된다.
- Step 목록이 많은 경우 스크롤 가능해야 한다.

제외:

- Step 로그 인라인 보기
- YAML 라인 점프

---

### 3.6 US-06. AI 분석

개발자로서, 워크플로우의 개선 포인트를 자동으로 제안받고 싶다.

가치:

- 사람이 놓치기 쉬운 성능, 보안, 구조 문제를 빠르게 파악할 수 있다.

세부 요구사항:

- 분석 대상은 현재 선택된 워크플로우 파일이다.
- 프론트는 분석 요청 전 대상 파일과 레포 메타정보를 구성한다.
- 백엔드는 Gemini 응답을 정규화한 JSON으로 반환한다.
- 프론트는 결과를 심각도 그룹으로 나눠 표시한다.
- 이슈에는 위치 정보, 설명, 제안이 포함되어야 한다.
- 분석 중에는 버튼이 중복 클릭되지 않아야 한다.

수용 기준:

- 분석 성공 시 이슈 목록이 표시된다.
- 분석 결과가 없으면 "문제 없음"이 아니라 "주요 이슈 없음"으로 표시한다.
- 분석 실패 시 오류 메시지와 재시도 UI를 제공한다.
- 분석 결과가 잘못된 스키마이면 파싱 실패로 처리한다.

제외:

- 워크플로우 자동 수정
- 개선 YAML 생성

---

## 4. 공통 UI 요구사항

### 4.1 상태

모든 주요 패널은 최소한 아래 상태를 가져야 한다.

- idle
- loading
- success
- empty
- error

### 4.2 접근성

- 상태 표현을 색상 하나에만 의존하지 않는다.
- 키보드로 워크플로우 목록과 실행 목록을 탐색할 수 있어야 한다.
- 주요 버튼은 로딩 중 비활성화된다.

### 4.3 반응형

- 데스크톱 우선으로 설계한다.
- 최소 1280px 기준 레이아웃을 최적화한다.
- 1024px 이하에서는 AnalysisPanel과 JobDetail을 탭 또는 아코디언으로 축소한다.

---

## 5. 비기능 요구사항

### 5.1 성능

- 첫 워크플로우 표시: 3초 이내
- 그래프 갱신: 1초 이내
- 분석 요청 응답: 15초 이내를 목표로 한다

### 5.2 보안

- PAT는 브라우저 메모리에서만 유지하고, sessionStorage/localStorage에 저장하지 않는다.
- Gemini API 키는 백엔드 환경 변수에만 둔다.
- 서버는 PAT를 저장하지 않는다.

### 5.3 안정성

- GitHub API 실패, YAML 파싱 실패, Gemini 실패는 각각 구분된 에러 상태를 가진다.
- 어떤 실패도 전체 앱 크래시로 이어지지 않아야 한다.

---

## 6. 제외 범위

MVP에서 제외한다.

- 다중 워크플로우 병합 그래프
- 규칙 기반 로컬 정적 분석 엔진
- 팀 협업 기능
- 사용자별 저장된 프로젝트 목록
- 다국어 UI

---

## 7. 출시 전 체크리스트

1. Public 레포 테스트 2개 이상
2. Private 레포 테스트 1개 이상
3. 성공/실패/진행중 run 표시 확인
4. AI 분석 실패 fallback 확인
5. 빈 워크플로우/잘못된 YAML 에러 확인
