# FlowPilot Agile Delivery Plan

## 1. 문서 목적

이 문서는 FlowPilot를 에자일하게 개발하기 위한 운영 계획을 정의한다.
핵심은 큰 설계를 한 번에 끝내는 것이 아니라, 작게 만들고 빠르게 확인하고 다음 우선순위를 조정하는 방식으로 MVP를 완성하는 것이다.

---

## 2. 에자일 개발 원칙

### 2.1 작은 단위로 자른다

- 기능은 1~3일 안에 끝낼 수 있는 단위로 나눈다.
- "컴포넌트 구현"보다 "public repo 연결 후 workflow 목록 표시"처럼 사용자 가치 기준으로 자른다.

### 2.2 동작하는 결과를 먼저 만든다

- 문서상 완벽함보다 실제 동작하는 수직 슬라이스를 우선한다.
- 프론트만 완성하거나 백엔드만 완성하는 방식보다, 얇더라도 끝까지 이어지는 흐름을 만든다.

### 2.3 우선순위는 고정하지 않는다

- 매 이터레이션 종료 시 우선순위를 다시 정한다.
- 사용성 문제나 기술 리스크가 보이면 다음 계획을 조정한다.

### 2.4 품질은 마지막 단계가 아니라 매 단계에 넣는다

- 각 단계마다 최소 테스트와 완료 기준을 함께 둔다.
- "나중에 정리"는 누적되면 속도를 오히려 떨어뜨린다.

---

## 3. 운영 단위

### 3.1 개발 cadence

권장 cadence:

- 1주 단위 iteration
- 하루 1회 진행 상황 점검
- iteration 종료 시 데모 + 회고 + 다음 계획 조정

현재 프로젝트 규모에서는 대규모 Scrum보다 가벼운 방식이 적합하다.

### 3.2 iteration 구성

각 iteration은 아래 4개로 운영한다.

1. 목표 설정
2. 작업 분해
3. 구현과 검증
4. 리뷰와 재계획

### 3.3 backlog 관리 기준

backlog는 아래 4개 상태로 관리한다.

1. `todo`
2. `in_progress`
3. `review`
4. `done`

규칙:

- 동시에 진행 중인 큰 작업은 1~2개로 제한한다.
- 미완료 작업은 다음 iteration으로 무조건 넘기기보다, 더 작은 단위로 다시 쪼갠다.

---

## 4. 단계별 개발 전략

### 4.1 Stage 0. 준비 단계

목표:

- 개발을 시작할 수 있는 최소 뼈대를 만든다.

핵심 질문:

- 워크스페이스 구조가 안정적인가
- 프론트와 백엔드를 같이 실행할 수 있는가
- 환경 변수 구조가 정리됐는가

작업:

1. pnpm workspace 설정
2. `frontend`, `backend` 생성
3. 공통 실행 스크립트 정리
4. 기본 lint/format/test 명령 정리

산출물:

- 모노레포 기본 구조
- 공통 실행 명령
- 기본 README

완료 기준:

- 로컬에서 프론트와 백엔드가 동시에 실행된다.
- 팀 없이 혼자 개발해도 실행 방법이 문서화되어 있다.

---

### 4.2 Stage 1. 레포 연결 슬라이스

목표:

- 사용자가 레포를 입력하고 workflow 파일 목록을 볼 수 있게 한다.

이 단계의 핵심은 "데이터가 실제로 들어오는가"를 확인하는 것이다.

작업:

1. Repo URL 파서 구현
2. PAT 메모리 보관 처리 및 public/private 전환 UX
3. GitHub repo 확인 API 연동
4. workflow 목록 조회
5. 첫 workflow 내용 fetch

산출물:

- RepositoryForm
- GitHub API client
- WorkflowSidebar 초안

완료 기준:

- public repo 1개 이상 연결 성공
- private repo 1개 이상 연결 성공
- `.github/workflows` 없을 때 empty state 표시

데모 질문:

1. 사용자가 레포 연결 흐름을 이해할 수 있는가
2. 에러 메시지가 다음 행동을 유도하는가

---

### 4.3 Stage 2. 구조 시각화 슬라이스

목표:

- 선택한 workflow가 DAG로 보이게 만든다.

이 단계의 핵심은 제품의 첫 번째 가치인 "한눈에 구조 파악"을 검증하는 것이다.

작업:

1. YAML parser 유틸 작성
2. jobs/needs 추출
3. graph transformer 작성
4. React Flow 그래프 렌더링
5. Job 선택 상태 연동

산출물:

- WorkflowGraph 모델
- GraphCanvas
- JobDetailPanel 기본판

완료 기준:

- 단일 Job, 선형 DAG, 병렬 DAG fixture에서 정상 동작
- 선택 노드 상태가 상세 패널에 반영
- 파싱 실패 YAML 에러 카드 표시

데모 질문:

1. YAML보다 빠르게 구조를 이해할 수 있는가
2. 그래프 레이아웃이 실제 사용에 충분한가

---

### 4.4 Stage 3. 실행 이력 슬라이스

목표:

- 구조가 아니라 실제 실행 결과까지 볼 수 있게 한다.

이 단계의 핵심은 제품의 두 번째 가치인 "어디서 실패했는가"를 검증하는 것이다.

작업:

1. 최근 workflow runs 조회
2. run 선택 UI 구현
3. run jobs 조회
4. graph 상태 merge
5. Step duration 표시

산출물:

- RunHistoryPanel
- Job execution merge 로직
- 상태 배지/색상 시스템

완료 기준:

- 실패 run 선택 시 실패 Job이 DAG에 반영
- JobDetail에 Step 실행 시간 표시
- 실행 이력이 없을 때 empty state 표시

데모 질문:

1. 실패 지점을 10초 안에 찾을 수 있는가
2. 상태 표현이 직관적인가

---

### 4.5 Stage 4. AI 분석 슬라이스

목표:

- 현재 선택한 workflow에 대한 AI 분석 결과를 제공한다.

이 단계의 핵심은 제품의 세 번째 가치인 "무엇을 고쳐야 하는가"를 검증하는 것이다.

작업:

1. NestJS analyze module 작성
2. Gemini service 작성
3. schema validation 추가
4. 프론트 분석 요청 연동
5. AnalysisPanel 구현

산출물:

- `POST /api/analyze`
- AnalysisPanel
- issue schema validator

완료 기준:

- 정상 결과가 severity별로 렌더링
- 잘못된 응답은 앱 크래시 없이 처리
- 재시도 UI 제공

데모 질문:

1. 제안이 실제 수정 행동으로 이어질 만큼 구체적인가
2. 허위 양성이 너무 많지 않은가

---

### 4.6 Stage 5. 안정화와 polish

목표:

- MVP를 실제로 써볼 수 있는 수준으로 다듬는다.

작업:

1. loading/empty/error 상태 정리
2. 테스트 fixture 추가
3. 주요 유닛 테스트 추가
4. 반응형/접근성 점검
5. 문서 정리

산출물:

- 기본 테스트 세트
- 안정된 UI 상태 처리
- 실행/배포 문서

완료 기준:

- 핵심 사용자 흐름이 끊기지 않는다
- 주요 실패 케이스에서 앱이 무너지지 않는다
- 첫 외부 사용 테스트가 가능하다

---

## 5. iteration 계획 예시

### 5.1 Iteration 1

목표:

- Stage 0 + Stage 1 완료

포함 작업:

2. 프론트/백엔드 실행 환경 구성
3. RepositoryForm
4. GitHub API client
5. workflow 목록 표시

iteration 결과:

- "레포 연결 -> workflow 목록 확인"이 동작

목표:

- Stage 2 완료

포함 작업:

1. YAML parser
2. graph transformer
3. React Flow 그래프
4. Job 상세 패널 기본판

iteration 결과:

- "workflow 선택 -> DAG 확인"이 동작

### 5.3 Iteration 3

목표:

- Stage 3 완료

포함 작업:

1. workflow runs 조회
2. run jobs 조회
3. 상태 오버레이
4. Step duration 표시

iteration 결과:

- "run 선택 -> 실패 지점 확인"이 동작

### 5.4 Iteration 4

목표:

- Stage 4 완료

포함 작업:

1. analyze API

### 5.5 Iteration 5

목표:

1. 에러/로딩 상태 정리
2. 테스트 보강

iteration 결과:

- MVP 배포 또는 내부 사용 시작 가능

## 6. 작업 단위 분해 규칙

### 6.1 좋은 작업 단위 기준

좋은 작업은 아래 조건을 만족한다.

1. 결과가 눈에 보인다
2. 하루 안에 끝낼 수 있다
3. 테스트하거나 확인할 방법이 있다
4. 다른 작업과 경계가 명확하다

예시:

- 좋은 작업: "repo URL 파서 구현 및 테스트 추가"
- 나쁜 작업: "프론트엔드 전체 구현"

### 6.2 PR 단위 기준

- 한 PR은 하나의 사용자 가치 또는 하나의 기술 리스크만 다룬다.
- PR이 너무 크면 리뷰가 아니라 통과 절차가 된다.

권장:

- 200~500라인 수준의 작은 PR
- UI 작업과 데이터 로직 작업이 과하게 섞이지 않게 유지

## 7. 완료 정의

### 7.1 Task Done

하나의 task는 아래를 만족해야 done으로 본다.

1. 코드가 동작한다
2. 최소 검증이 끝났다
3. 오류 상태를 고려했다
4. 다음 작업자가 이해할 수 있다

### 7.2 Stage Done

한 stage는 아래를 만족해야 완료로 본다.

1. 사용자 관점의 데모가 가능하다
2. 문서상의 완료 기준을 만족한다
3. 다음 stage로 넘어갈 때 큰 가정이 남아 있지 않다

### 7.3 MVP Done

MVP는 아래를 만족해야 한다.

1. public/private repo 연결이 가능하다
2. workflow DAG가 보인다
3. 최근 실행 결과가 보인다
4. AI 분석 결과가 안정적으로 보인다
5. 치명적인 크래시 없이 기본 에러 처리가 된다

---

## 8. 회고와 우선순위 조정

각 iteration 끝에 아래를 기록한다.

1. 무엇이 계획보다 오래 걸렸는가
2. 어떤 기술 가정이 틀렸는가
3. 사용자가 가장 먼저 가치 있다고 느낄 부분은 무엇인가
4. 다음 iteration에서 줄여야 할 범위는 무엇인가

회고 결과에 따라 해야 할 조정 예시:

- YAML 뷰어가 계속 뒤로 밀리면 MVP 제외를 확정
- AI 품질이 낮으면 rule-based check를 먼저 넣는 방향 검토
- GitHub API 매칭 문제가 크면 run overlay 범위를 축소

---

## 9. 리스크 기반 우선순위

아래 순서로 검증하는 것이 적절하다.

1. GitHub 레포 연결이 안정적인가
2. YAML에서 DAG를 안정적으로 만들 수 있는가
3. run 결과를 graph에 무리 없이 입힐 수 있는가
4. Gemini 결과를 정규화할 수 있는가

이 순서가 중요한 이유:

- 앞단이 불안정하면 뒤 기능은 의미가 없다.
- AI는 제품 차별점이지만, 기본 데이터 흐름이 먼저 안정적이어야 한다.

---

## 10. 권장 운영 방식

현재 프로젝트는 1인 또는 소수 인원 개발에 가깝다.
따라서 무거운 프로세스보다 아래 운영 방식을 권장한다.

1. 월요일 또는 시작일에 iteration scope 확정
2. 매일 짧게 현재 blocker 기록
3. 기능 하나 끝날 때마다 바로 문서/스크린샷 업데이트
4. iteration 마지막 날에는 데모 기준으로 점검

즉, 이 프로젝트의 에자일 핵심은 "회의"가 아니라 "작은 완료 단위와 빠른 재계획"이다.
